# 智能电商运营 Agent 平台设计

## 背景与目标

当前仓库来自企业知识库问答系统，已经具备 FastAPI、Vue 3、登录权限、RAG、LangChain、Docker、测试和生产配置等工程底座。本项目目标不是在原系统中附加一个电商页面，而是把复制后的新仓库重新命名并替换成一个专门的“智能电商运营 Agent 平台”。

项目需要在简历和演示中突出两类能力：

- AI Agent 工程能力：任务规划、工具调用、Agent 推理过程、结构化建议、人工审批闭环、可审计执行记录。
- 数据分析能力：基于模拟电商经营数据完成指标计算、异常检测、归因分析、商品分层、ROI 和库存风险判断。

第一版必须在没有真实电商平台和内部业务系统的情况下可完整演示。系统使用内置模拟数据和规则分析引擎作为稳定默认能力；如果配置了大模型 API Key，则在同一套接口上增强自然语言解释和策略生成。

## 产品定位

产品名称：智能电商运营 Agent 平台。

产品形态：一个面向电商运营团队的 AI 工作台，包含运营驾驶舱、Agent 对话分析、商品运营、活动策略、建议审批和运营知识库。

简历项目表述：

> 基于 FastAPI、Vue 3、LangChain 和 RAG 构建智能电商运营 Agent 平台，设计模拟订单、流量、广告、库存、评价数据集，实现 GMV、转化率、ROI、库存风险和评价异常的自动归因分析，并通过多工具调用 Agent 生成运营建议、活动方案和审批任务，形成从数据分析到运营动作建议的闭环。

## 替换边界

保留工程底座：

- FastAPI 应用结构、路由注册方式、中间件、CORS、健康检查。
- Vue 3、Element Plus、Pinia、Vue Router、Vite 前端结构。
- 登录、权限、管理后台、API client 的基础模式。
- 现有 RAG、LangChain、文档处理和评测能力的可复用代码。
- Docker Compose、测试目录、CI 和生产配置。

替换产品表达：

- 企业知识库问答系统改为智能电商运营 Agent 平台。
- 智能问答改为运营 Agent 工作台。
- 文档管理改为运营知识库。
- RAG 自动评测改为 Agent 分析质量评测。
- 管理后台改为运营管理后台。
- README、页面标题、侧边栏、健康检查、OpenAPI 标题和默认导出文件名全部改为电商运营语义。

弱化或重命名旧能力：

- 原知识库能力保留为“运营知识库”，服务商品资料、品牌规则、活动复盘和客服 SOP。
- 原聊天能力重命名为“运营 Agent”，接入电商分析工具，而不是只做通用问答。

## 核心用户流程

1. 用户登录系统后进入电商运营驾驶舱。
2. 驾驶舱展示 GMV、订单数、转化率、客单价、广告 ROI、库存风险、差评率等核心指标。
3. 系统基于模拟数据自动生成异常提醒，例如“主推商品转化率下降”“某 SKU 库存低于安全线”“广告 ROI 低于阈值”。
4. 用户进入运营 Agent 工作台，提出问题，例如“昨天 GMV 为什么下降”“本周哪些商品适合做大促”“哪些广告计划应该降预算”。
5. Agent 根据问题选择工具，查询模拟经营数据，执行指标计算和归因分析。
6. Agent 输出结构化结论，包含关键发现、数据证据、原因判断、建议动作、风险等级和置信度。
7. 高风险建议进入审批任务列表，例如改价、加预算、发券、清仓。
8. 用户在审批页查看建议依据，选择通过、拒绝或标记为已执行。
9. 系统保留执行记录，用于演示企业级治理和可追溯性。

## 后端设计

新增业务包：

- `backend/ecommerce/`
  - `data_loader.py`：加载模拟 CSV/JSON 数据。
  - `metrics.py`：计算 GMV、转化率、客单价、ROI、退款率、库存周转等指标。
  - `anomaly.py`：识别销售、流量、广告、库存、评价异常。
  - `segmentation.py`：商品分层，例如主推款、利润款、引流款、清仓款、风险款。
  - `tools.py`：定义 Agent 可调用的经营分析工具。
  - `agent.py`：根据用户意图编排工具调用，生成结构化分析结果。
  - `schemas.py`：定义响应模型和审批任务模型。

新增 API 路由：

- `GET /api/ecommerce/dashboard`：返回运营驾驶舱指标、趋势和异常列表。
- `GET /api/ecommerce/products`：返回商品分层、销售表现和库存风险。
- `POST /api/ecommerce/agent/analyze`：接收运营问题，返回 Agent 工具调用轨迹和分析结果。
- `GET /api/ecommerce/recommendations`：返回待审批运营建议。
- `POST /api/ecommerce/recommendations/{id}/approve`：审批建议。
- `POST /api/ecommerce/recommendations/{id}/reject`：拒绝建议。
- `GET /api/ecommerce/campaigns/plan`：生成活动选品和策略建议。

Agent 默认采用确定性规则编排，确保无 API Key 时可演示。可选 LLM 增强模式通过现有 LLM gateway 生成更自然的解释，但不能改变底层指标计算结果。

## 模拟数据设计

新增目录：

- `data/ecommerce/products.csv`
- `data/ecommerce/orders.csv`
- `data/ecommerce/traffic.csv`
- `data/ecommerce/ad_spend.csv`
- `data/ecommerce/inventory.csv`
- `data/ecommerce/reviews.csv`
- `data/ecommerce/competitors.csv`
- `data/ecommerce/operation_rules.json`

数据覆盖 30 到 90 天的经营周期，并内置可解释异常场景：

- 主推商品流量稳定但转化下降。
- 广告花费增长但 ROI 下降。
- 某 SKU 库存接近安全线。
- 新品点击率高但详情页转化弱。
- 某商品差评率上升并影响转化。
- 竞品降价导致价格敏感商品销量下滑。

这些异常必须能被后端规则复现，避免演示时依赖随机结果。

## 数据分析能力

第一版实现以下分析能力：

- 指标总览：GMV、订单数、销量、客单价、转化率、退款率、广告花费、广告 ROI。
- 环比分析：与前一日、前 7 日均值或上一周期对比。
- 商品分析：销售贡献、利润贡献、转化效率、库存风险、差评风险。
- 广告分析：花费、成交、ROI、低效计划识别。
- 异常检测：基于阈值、环比变化和业务规则识别异常。
- 归因输出：把指标变化拆成流量、转化率、客单价、退款、广告、库存和评价因素。
- 行动建议：改标题、优化详情页、降预算、补库存、清仓、发券、客服跟进。

每个分析结果都要返回证据字段，包含相关指标、对比基准和触发规则，便于前端展示“Agent 为什么这么判断”。

## AI Agent 能力设计

Agent 不只输出文本，而是返回可视化的执行过程：

- 用户问题。
- 意图识别结果，例如经营诊断、商品优化、活动规划、广告复盘。
- 工具调用轨迹，例如 `get_sales_summary`、`detect_anomalies`、`rank_products`、`analyze_ad_roi`。
- 数据证据。
- 结论摘要。
- 建议动作。
- 风险等级。
- 置信度。
- 是否生成审批任务。

工具示例：

- `get_kpi_snapshot(date_range)`
- `compare_metric(metric, current_range, baseline_range)`
- `detect_sales_anomalies()`
- `rank_products_by_contribution()`
- `find_low_roi_ads()`
- `find_inventory_risks()`
- `summarize_review_issues()`
- `generate_campaign_plan(goal)`

Agent 需要支持几类内置问题模板：

- “昨天 GMV 为什么下降？”
- “哪些商品适合参加大促？”
- “哪些广告计划 ROI 太低？”
- “哪些商品存在库存风险？”
- “帮我生成本周运营日报。”

## 前端设计

整体前端从知识库工作台替换为电商运营工作台。

页面结构：

- `/dashboard`：运营驾驶舱，作为登录后的主要页面。
- `/agent`：运营 Agent 工作台，展示对话、工具调用轨迹、分析结论和建议。
- `/products`：商品运营分析，展示商品分层、核心指标和风险标签。
- `/campaigns`：活动策略中心，生成选品、优惠和投放建议。
- `/recommendations`：建议审批中心，展示待审批、已通过、已拒绝、已执行建议。
- `/knowledge`：运营知识库，复用原文档上传和检索能力。
- `/evaluation`：Agent 分析质量评测，复用原评测底座。
- `/admin`：运营管理后台。

视觉风格应偏运营工具，不做营销落地页。界面以高密度数据、表格、趋势图、状态标签和任务列表为主，突出专业感和可演示性。

## 审批与治理

建议动作分为三类：

- 低风险：文案优化、标题建议、客服话术，直接展示。
- 中风险：活动方案、优惠建议、商品分层调整，进入待确认。
- 高风险：改价、加广告预算、发券、清仓，必须进入审批中心。

审批记录包含：

- 建议标题。
- 触发原因。
- 数据证据。
- 风险等级。
- 预期影响。
- 当前状态。
- 操作人。
- 更新时间。

第一版审批状态保存在内存或本地 JSON 中，后续可迁移到数据库。

## 错误处理

- 模拟数据缺失时返回明确错误，并提示缺失文件路径。
- 指标计算遇到除零时返回 0 或 `null`，并在证据中说明样本不足。
- LLM 调用失败时降级为规则解释，不影响核心分析。
- Agent 无法识别问题时返回可选问题模板，而不是空回答。
- 审批任务不存在或状态非法时返回 400/404。

## 非目标

第一版不接入真实淘宝、天猫、抖店、京东或 Shopify API。

第一版不实现真实改价、真实投放、真实发券。

第一版不追求复杂 AutoGPT 式自主循环，避免不可控和难演示。

第一版不把所有原知识库功能删除，保留可服务运营知识库的基础能力。

## 实施顺序

1. 替换项目命名、README、健康检查、前端品牌和路由标题。
2. 添加模拟电商数据集。
3. 实现后端电商指标、异常检测和商品分层。
4. 实现 Agent 工具注册、工具调用轨迹和结构化分析响应。
5. 实现建议审批数据模型和 API。
6. 改造前端为运营驾驶舱、Agent 工作台、商品分析和审批中心。
7. 添加测试和演示说明。

## 成功标准

项目完成后，用户可以把该仓库作为独立的“智能电商运营 Agent 平台”展示，而不是企业知识库问答系统的换皮页面。

系统在无真实平台、无内部资料、无 API Key 的情况下仍能完成端到端演示；在配置 LLM 后，可以增强自然语言分析，但核心数据分析和 Agent 工具调用能力不依赖外部服务。
